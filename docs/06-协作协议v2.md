# 06. 协作协议 v2 + 安全与可观测加固（2026-06-11）

## 一句话定位

这一阶段，AgentBridge 的协作机制从 v1 的「提示词契约」（靠 `⏳/✅` 文本暗号 + 注释维系的隐式状态）正式升级为 v2 的「协议级状态契约」——turn 生命周期变成可观测、可关联的结构化状态机；与此同时，借公开发布的契机一次性偿还了安全面与可观测性的工程债。一句话：**协作从「约定俗成」走向「协议定义」，安全从「无人把守」走向「准入校验」。**

## 背景

v1 把 Claude 和 Codex 接上了线，但协作的「turn 协调」是一套软启发式：Codex 在干活时桥发一行 `⏳ Codex is working` 给 Claude 看，干完发 `✅ Codex finished`，Claude 凭这两个文本标记决定能不能发 `reply`。这套机制在简单的「一问一答」里够用，但在架构审视（见上一阶段）里，Claude 和 Codex 各自独立得出了同一个最强共识：**文本暗号撑不住复杂场景**。

具体痛点有两层：

- **状态轴混淆**：「Codex 这个 turn 在不在跑」和「桥的出站窗口开没开」是两件不同的事，但都被压进同一行文本里，消费者无从区分一个 turn 是正常运行、被 watchdog 判定卡住、还是异常中止。Claude 发出一次 `reply` 后，也没有任何关联信息能告诉它「是哪一次 reply 开启了哪一个 turn」——无法做幂等、无法做超时归因。
- **安全面敞口**：daemon 的控制 WebSocket（:4502）和 Codex 代理（:4501）在 v1 里基本是「本机谁连谁就是合法的」。公开发布后，本机上任意网页（通过浏览器）都可能对 `ws://127.0.0.1:4502` 发起跨站 WebSocket 连接（CSWSH），从而注入指令给 Codex、或读取 Codex 的输出。这条攻击面必须在发布前收口。

加上 v1 遗留的一批工程债——进程存活判定散落多处、Codex→Claude 的投递路由是隐式分支、`daemon.pid` 和 `status.json` 两个文件容易写到不一致——这一天（v0.1.7~0.1.13，密集发布）把这些一并处理掉。

## 做了什么

### 1. turnPhase 状态契约（协作协议 v2 PR A）

把「Codex turn 在干什么」从一行文本提升为 `DaemonStatus` 上的正式字段：

```
turnPhase: "idle" | "running" | "stalled" | "aborted"
```

四个值**严格只表达 Codex turn 的生命周期**：`running` 是正在跑；`stalled` 是 watchdog 判定一段时间无活动（活动一恢复就退回 `running`）；`aborted` 是最近一个 turn 异常结束且其后没有新 turn 接上；`idle` 是没有活跃 turn。

关键的设计纪律是**轴的分离**：桥自己的「路由/出站窗口」用一个**独立字段** `attentionWindowActive` 表达，刻意不混入 `turnPhase`。同时为了不打破旧消费者，`turnInProgress` 兼容保留一个版本，语义固定为 `turnPhase ∈ {running, stalled}` 的映射。一致性要求是硬约束：`/healthz`、`status.json`、控制端 `status` 三个 payload 的这些字段必须一致，turn 转移和 attention 转移都会去刷新 `status.json`（写失败只记录、不阻断核心路径）。

### 2. steer：把消息喂进运行中的 turn（PR B0）

`reply` 工具新增 `on_busy` 参数。在此之前，Codex 忙时 `reply` 只能被拒（busy guard）。B0 引入 `on_busy: "steer"`：在 Codex turn 运行中，经 app-server 的 `turn/steer` 把 Claude 的消息**喂进当前 turn**——不新开 turn、不打断、不丢弃已有工作，适合「中途修正」。daemon 侧统一加 `[STEER from Claude]` 前缀，让 Codex 能区分「mid-turn 更新」和「原始任务指令」。

实现里踩到一个真机 E2E bug：`turn/steer` 从 codex-rs v0.99 起就要求必填 `expectedTurnId`（当前活跃 turn 的 id），缺失或不匹配直接被拒。修复方式是让 codex-adapter 从 `turn/started` 事件持续跟踪活跃 turn id 并自动填充；若没有可寻址的 turn id，steer 在本地就 transport-reject、不发往 app-server。steer 被 app-server 拒绝（各种 mismatch / NoActiveTurn race）**不算 turn 终结**——不发 `turnAborted`、不改 `turnPhase`，而是以 `system_steer_failed` 系统消息显式告诉 Claude「消息没送进去，原 turn 不受影响」。

### 3. interrupt + ACK + 注入幂等（PR B）

PR B 是协议的核心，补齐了「打断」和「可关联」两块：

- **打断入口 = `reply` 的 `on_busy: "interrupt"`**：与 steer 相反，interrupt 经 app-server 的 `turn/interrupt` **终结当前 turn**，然后用 Claude 的新消息开一个新 turn——只在「当前工作已经过时」时用。
- **`turn_started` ACK**：新增独立的控制事件 `turn_started`，携带 `requestId / idempotencyKey / threadId / turnId` 关联字段。没有这些字段，Claude 根本无法判断「是哪一次 reply 开启了哪一个 turn」。`claude_to_codex_result` 的语义随之收窄为 **accepted**（daemon 已接收并尝试写 `turn/start`），Claude 的 15s 超时只适用于「accepted 都没返回」。
- **幂等键状态机**：daemon/codex-adapter 侧维护 `(threadId, idempotencyKey): accepted → started(turnId) → terminal`。terminal（completed/aborted/rejected）后保留 **TTL 20 分钟的 tombstone**——否则一个快速完成的 turn 会让迟到的重试在 terminal 之后再次注入，幂等就破功了。
- **结构化 result**：`{ok:false, code, phase?, retryAfterMs?}`，code 是一份明确清单（`busy_reject / interrupt_timeout / duplicate_in_flight / ...`），旧 `error` 字符串并存一个版本。

### 4. 安全三连

- **Origin guard 防 CSWSH**：WebSocket upgrade 阶段校验 `Origin`，阻断本机网页对控制端口和 Codex 代理的跨站连接——堵死「网页注入/读取 Codex」这条攻击面。
- **attach 守卫 + 能力令牌**：`claude_to_codex` 注入加 attach 守卫——只有真正完成 attach 的合法会话才能注入；控制端引入能力令牌。
- **契约版本协商**：控制协议的 attach 准入会校验 `contractVersion`，不匹配直接以 CLOSE 码 4006 拒绝——避免新旧 daemon/foreground 在协议演进时静默错配。
- **TOCTOU 补丁**：interrupt 路径的 attach 守卫存在 time-of-check-to-time-of-use 窗口——interrupt 要 `await` 等待 turn 进入 terminal，这段 await 期间授权状态可能变化。修复是在 **await 之后重新校验注入授权**，而不是只在进入时查一次。

### 5. abg resume / abg logs（故障恢复最后一公里）

- **`abg resume`**：一键恢复双侧会话——这正好对接了「额度守卫」体系里 Claude 被打到硬线后由 Codex 接棒、之后再恢复的场景。
- **`abg logs`**：新增日志命令，把诊断信息从「需要自己去翻 state dir 下的 `agentbridge.log`」降级为一条命令，补上故障恢复的最后一公里。
- `abg claude` / `abg codex` 默认以最高权限启动，省掉每次手动放权的摩擦。

### 6. 架构 refactor

- **进程身份与存活判定收敛**到单一模块 `process-lifecycle`——此前散落在多处的「这个 pid 还活着吗 / 这是不是我们的进程」判断统一口径。
- **Codex→Claude 投递路由显式化**为纯函数 `routeCodexMessage`——把原先隐式的 if/else 路由分支抽成一个可单测的纯函数（投递路由矩阵）。
- **抽出 `BoundedMessageBuffer` / `PendingRequestRegistry`**——有界消息缓冲和挂起请求登记表独立成模块。
- **`daemon.pid` + `status.json` 合并为单一原子 `daemon.json`**——消除两个文件写到不一致的窗口。
- **Codex→Claude 投递去重**：稳定的 `message_id` + 消费端 LRU/TTL deduper，防止重复投递。
- **测试按进程边界分层**：集成测试移入 `src/integration-test/`，与纯逻辑单测分开。

## 关键设计决策与为什么

**turnPhase 为何要做成正式的状态契约，而不是继续用文本？** 因为协作链路要从「文本暗号 + 注释维系的隐式状态」升级为「结构化、可观测、可关联」。文本暗号只能被人/agent「读到」，无法被程序「消费」——做不了幂等、做不了超时归因、做不了多 payload 一致性校验。把 turn 生命周期固化成枚举字段，并刻意把「桥的出站窗口」分到 `attentionWindowActive` 这个**独立轴**，是为了让每个轴的语义单一、可独立演进；混在一起的话，消费者永远在猜「这个状态到底说的是 turn 还是窗口」。

**steer 和 interrupt 的语义区别为何要分成两个 `on_busy` 值？** 二者在「当前工作还要不要」上是相反的：steer 是**保留**当前 turn、把新信息喂进去（中途修正，不丢已有进度）；interrupt 是**放弃**当前 turn、用新消息重开（当前工作已过时才用）。把它们做成 `reply` 的同一个参数的两个取值（而非两个工具），是因为它们共享完全相同的「busy 时如何处置」决策点，只是处置方向不同——调用方在同一个地方做二选一，语义最清晰。

**attach 守卫的 TOCTOU 为何必须 await 后重检？** interrupt 不是瞬时操作：它要等被打断的 turn 真正进入 terminal（codex-rs 的实证是 interrupt 的成功响应会被 app-server 推迟到 `TurnAborted` 才回）。这意味着「检查授权」和「实际注入新 turn」之间隔着一段不确定时长的 `await`。如果只在进入时查一次授权，await 期间会话若已失去授权（比如另一个会话抢了 attach），就会出现「检查时合法、使用时已非法」的注入。修复就是把授权校验挪到 await **之后**、真正注入**之前**——这是安全守卫面对异步等待时的标准做法。

**Codex-skill 集成为何留作 draft、没在本阶段落地？** 这个设计（把协作协议做成 Codex 原生 skill，装到 `~/.agents/skills/`，让 Codex 启动期自动发现，根治「Codex 反复在源码里找不存在的 send-to-Claude API」的问题）方向是对的，但它的 scope 与本阶段不同：协议 v2 是**桥内部的协议演进**（per-pair、运行时行为），而 skill 集成是**机器级的安装机制**（跨 harness、改 CLI 的 init/dev 流程、新增 uninstall 命令、还有「装哪个目录 / per-project 还是 machine-level」等悬而未决的 open question）。把它和安全/协议加固混在同一批 commit 里会污染 reviewer 的验收边界，也会让「协议契约」这件事失焦，因此固化为设计稿、单独排期。draft 里的 P1（修 `collaboration-content.ts` 的 AGENTS.md 段、补「透明代理」说明）是低风险的文档修复，可随时单独做。

## 产出

- **协议级协作契约**：turn 生命周期成为可观测、可关联的状态机（turnPhase 四态 + attentionWindowActive 独立轴 + `turn_started` ACK + 幂等键状态机），steer/interrupt 给 Claude 提供了「忙时」的两种明确处置手段。
- **安全基线**：Origin guard（防 CSWSH）+ attach 守卫（含 TOCTOU 修复）+ 控制端能力令牌 + 契约版本协商（CLOSE 4006）——发布前把控制端口和 Codex 代理的攻击面收口。
- **更干净的架构模块**：进程存活判定单一化（`process-lifecycle`）、投递路由纯函数化（`routeCodexMessage`）、状态文件原子化（`daemon.json`）、投递去重（稳定 message_id + LRU/TTL deduper），以及测试按进程边界分层。
- **故障恢复闭环**：`abg resume` 一键恢复双侧会话、`abg logs` 一键取诊断，配合默认最高权限启动。
