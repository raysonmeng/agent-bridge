# 07. 额度协调与策略 v3（Bridge × QuotaGuard）（2026-06-10 ~ 06-18）

## 一句话定位

这九天，AgentBridge 从一个"会把消息从 Claude 转给 Codex"的协作工具，长出了一层**会自己管订阅额度的智能**：它持续盯着两个 AI 账号各自的 5h / 周窗口用量，算出"现在该谁多干、能不能开新任务、还剩多久会撞限流"，在周窗口刷新前把额度尽量用满到逼近 98%、却又不让进行中的任务被腰斩；万一真被外层硬线停住，窗口刷新后还能在原本的 TUI 里**全自动续上**没做完的活。这是整个项目投入最大的一条线——从 v0.1.14 到 v0.1.22，二十多个 PR，跨 `agent_bridge` 和 `agent-quota-guard` 两个仓库协同。

## 背景：订阅额度是会"作废"的，长任务会被它卡死

Claude 和 Codex 都是订阅制，额度按**滑动窗口**结算——一个 5 小时短窗口、一个 7 天周窗口。这带来两个互相矛盾的痛点：

- **用不满就是浪费。** 周窗口结束时没用完的额度直接清零、不结转。理想状态是在周刷新前一刻把用量推到接近 100%，但现实里没人会精确盯着窗口，往往一周结束还剩一大截没用——白白浪费了订阅费。
- **用过头就被腰斩。** 反过来，如果一个长任务一路冲到供应商硬限流，正在跑的工作会被中途打断，已经投进去的 token 全打水漂。

老的 v2.x 协调器是"保守防超"哲学：任一侧用量 `gateUtil ≥ 90%` 就一刀切暂停，降到 30% 以下才解除。它能防爆雷，但和"用满"的目标正面冲突。设计稿里记了一个**真实反面案例**（2026-06-11 实拍）：Codex 周窗口用到 92%，但 **6.5 小时后（半夜）就会刷新**，5h 窗口才用了 22%——老逻辑直接 `92 ≥ 90` 关闸、拒绝一切委派，把刷新前剩的 8% 周额度白白冻结了 6.5 小时。这正是 v3 要解决的第一个验收用例。

更深一层的问题是：probe 探针**只给百分比，不给绝对 token 数**。Claude 是最顶级套餐、Codex 是次顶级套餐，同样 80% 的绝对余量差很多。光看百分比之差来分配任务（v2 的做法）是错的。

## 做了什么

这条线分成几大块，按"地基 → 策略 → 续命 → 打磨"的顺序铺开：

**① Bridge × QuotaGuard 协调层（P0–P4，#97，06-10/11）。** daemon 周期性调用已部署的 quota-guard 探针，并发拿到双方的 5h / 周用量，算出 `normal / balance / parallel / paused` 的联合决策，通过 `DaemonStatus.budget`、`get_budget` MCP 工具、`abg budget` 三个界面暴露；`paused` 时对 `claude_to_codex` 注入口加硬闸门。这是把"额度感知"接进 bridge 的骨架。

**② budget 策略 v3 的四个台阶**——这是本阶段的核心，每个台阶都是独立 PR、独立验收：

- **P1 燃尽率（#160，地基）**：展示"按当前消耗速度，每侧还能再工作几小时"（runway），以及周窗口刷新时会用到百分之多少。关键决策是**采集口径下沉到 quota-guard**（见下文）。
- **P2 时间感知暂停线 / maximize（#174）**：把"暂停判据"从常数阈值翻转成"离窗口刷新还有多久"的函数（详见下文）。
- **P3 三态 admission 闸门（M1–M3b，#180/#181）**：把闸门从 open/closed 二态升级成三态，区分"拒新任务"和"放收尾"（详见下文）。
- **P4 runway 分配（#178）**：把任务分配判据从"用量百分比之差"升级成"剩余可工作时间之差"，并新增"烧不满"时的加速建议。

**③ 全自动续接（PR2–PR5，#165–#168，06-14）**：当某一侧被额度硬线停在干净点（轮末）并写下 checkpoint 后，bridge 检测到该侧窗口刷新，自动把"从 checkpoint 下一步继续"的任务注入回原本的 TUI——Codex 侧靠 `turn/start` 开新 turn，Claude 侧靠 channel push + ack/重试/降级。这是 guard 仓和 bridge 仓**跨两个代码库协同**完成的。

**④ 收尾打磨（06-16~06-18）**：v3.2 把策略简化成"只有动态线、逼近 98"（#176）；共享额度节奏提示词（#179）；额度时间一律渲染北京时区（#182）；`get_budget` 过期即实时刷新 + 空闲时不刷屏（#184）。

## 关键设计决策与为什么

### 时间感知动态暂停线 + maximize 逼近 98%

这是整条线最核心的翻转。直觉很简单：**离窗口刷新越近，"现在多烧一点"的代价越低**（反正马上刷新归零），可容忍的用量越高；离刷新越远，提前打满的代价是长达数天的瘫痪期，必须留缓冲。

于是暂停线不再是常数 90，而是 `time-to-reset` 的函数，且**按 5h 和周两个窗口分别计算**（各有自己的重置时间和燃尽率）。核心逻辑（`dynamicPauseAt`）：先用"当前用量 + 燃尽率 × 距刷新小时数"预测刷新时的用量——

- 如果**预测烧不满**目标线，就放行（不暂停），甚至建议加速多派任务；
- 如果**预测会烧满**，暂停线 = 目标 − 收尾余量 − 时间缓冲（缓冲随距刷新时间线性增长，远期顶到上限后退化回老的 90，跟 v2 等价）。

回到那个 92% 的真实案例：预测刷新时 ≈ 99.8% > 98%，走"会烧满"分支，但算出的线是 93.4 > 当前 92 → **不暂停、继续干**——正确地把那 8% 用掉了。而"最后 1h 用到 96%"这种会撞穿目标的场景，线压到 95.6，正确触发暂停。

**v3.2 的简化（#176）很关键**：用户明确要求"默认就用满、别再给我 conserve/maximize 二选一、别设 92/97 这种限制、写死逼近 99 再停"。这里有个澄清——"逼近 99 再停"**正是动态线的渐近语义**（线随刷新临近而升），不是 `util ≥ 99` 的死阈值（那会在离刷新很远时就早早烧满、然后冻结好几天）。所以删的是**策略选择和过度配置**，不是 pacing 本身：删掉 `strategy` 字段，决策恒走动态线；`targetUtil` 定 98（留 2% 给供应商计量抖动，比 99 稳）；老配置里残留的 `strategy` 键被容忍忽略。这条改动是 `[skip release]` 的纯内部演进。

配套两条不变量贯穿始终：**I1（只放宽不收紧）**——maximize 的任何暂停线都不低于老的 90，绝不会比 v2 更早停；**I2（phantom-hold）**——曾进入暂停的窗口，在探针数据变 stale 期间维持原状态，**绝不因数据缺失而开闸**。

### 三态 admission 闸门：open / admission-closed / closed

老闸门只有 open/closed 二态：关闸 = 拒绝一切注入。但这和"不打断进行中任务"的宗旨冲突——一个多轮协作任务的**收尾**（验收回包、写 checkpoint）通常还要 1–2 个短 turn，二态闸门把这些也拒了。

v3 引入中间态 `admission-closed`（5h 用量进入收尾保护带时触发）：

- **拒绝新任务** `turn/start`（新错误码 `budget_admission`，文案说明"5h 收尾保护中，仅接收收尾类注入"）；
- **放行 steer 注入**（喂给正在跑的 turn 的中途修正，中断它反而浪费已投入的 token）；
- **放行带 `wrap_up: true` 标记的收尾 reply**——但有防滥用护栏：每个 5h 窗口最多放行 `wrapUpQuota`（默认 2）个收尾 turn，且**配额消耗持久化**到磁盘（`admission-quota.json`），防 daemon 重启清零变成绕闸后门；
- 进行中的 turn **永不被闸门打断**（闸门只挡注入口）。

`closed`（真没额度了）态下还有一个 Codex 对抗审挖出的关键补丁（REAL-2）——**system-initiated checkpoint baton**：closed 触发的瞬间，若 Codex idle 且本窗口没用过，放行**恰好一个**系统发起的小型 checkpoint turn。动机就是避免复现那个反面案例——动态线触发的一刹那连"写 checkpoint 收尾"都做不了，把任务冻死在半路。

M3b 这层（#180）的真机验收覆盖了六个场景，其中两个是纯靠代码 review 难发现的时序坑：admission directive 在 Codex turn 进行中要**延迟到 turn 结束才发**（不能 mid-turn 刷给 Claude）；以及 `on_busy: "interrupt"` 路径在 await 中断边界后**必须重新检查闸门**——否则等待期间用量涨过线，注入会绕闸成功。这两条都是 Codex 跨引擎 review 时钉死的 gate-bypass 漏洞。

### 燃尽率为什么下沉到 quota-guard（fresh-if-stale 的前身）

最初的设计是 bridge 自己采样、自己算 EWMA 燃尽率。06-11 用户提出、双方确认了一个**分层修正案**（#162）：采集 / EWMA / 置信判定 / 持久化**全部下沉到 agent-quota-guard**，bridge 退化成纯消费端。原因很硬：guard 已经拥有探针数据源、缓存和历史文件，在 bridge 侧重复实现就是**两套估计器**——一处口径漂移就会出现两个互相矛盾的"还能干多久"。

落地的契约是 probe 输出可选追加 `burn_rate_pct_per_hour / burn_confident / runway_seconds / depleted_at_epoch` 四个字段。bridge 侧只做三件事：**严格校验透传**（字段非数值/负值/NaN 就整组丢弃，但窗口本身保留）、**跨窗口取最小 runway 的选择**、**时长格式化**。bridge **禁止自己重算 burn-rate**——这是 Codex 的硬验收约束。结果是 bridge 删掉了约 460 行采样/EWMA/持久化代码。

### 全自动续接：guard × bridge 跨仓协同

"停了要能自动续命"分成两仓的职责：

- **guard 侧（减速线，remind 不 deny）**：去掉中途硬线 deny，改成强提醒（执行工具调用中途**绝不拦**）；轮末到硬线时**停在干净点 + 写 checkpoint + 写 pending 文件**（这个"停"是续接的前提，必须保留）。
- **bridge 侧（检测刷新 + 注入续接）**：检测到"曾暂停的某侧窗口刷新了"，读 guard 写的 pending，把续接任务注入回 TUI。

最棘手的是 Claude 侧——官方 issue #44380 说 idle 时 channel 消息只显示、Claude 不自动处理。但 06-14 的**本机实测推翻了这个悲观判断**：Claude 真 idle（甚至明确收尾、空闲 3 分钟）时，bridge channel push 一条消息，Claude 立即自动响应、用户全程没碰键盘。结论是本机环境下 Claude 侧续接是 fully automatic，但保留 `ack_resume` + 少量重试兜底（不是因为唤不醒，是防 #61797 报过的偶发丢消息），连续失败 N 次才降级到 SessionStart。

续接的**幂等身份**做得很硬（#175）：去重身份 = `sha256(agent \0 sessionId \0 realpath(cwd) \0 contentHash)`，落盘在全机共享的 `~/.budget-guard/{claims,consumed}/`，靠**文件名即 identity** 隔离多 pair。`claims/` 是 `O_EXCL` 写的在途锁（保证跨进程同一身份只有一个在途注入），`consumed/` 是幂等墓碑（**必须跨重启/跨 kill 存活**，否则 `abg kill` 重开会把已消费的续接重新注入）。`abg kill` 有意**不清**这两个目录，靠 TTL prune（consumed 7 天 / claims ~320s 孤儿）回收。

Codex 侧续接注入还修了一个时序竞争（#172）：原本确认 turn 开始后立刻推进队列，与刚开的 turn 竞争（codex-rs 不保证 `turn/start` 响应早于 `turn/started` 通知）。**改成 drain-only**——确认时只 consume/delete、不推进，下一条 pending 只在 terminal/drain 边界推进，时序对正确性不再相关。

### 为什么 fresh-if-stale（#184，收尾的点睛）

部署后用户碰到两个真实问题：① 空闲的 pair 仍被 balance/欠载通知刷屏；② **真正做任务分配决策时，`get_budget` 返回的是上一次轮询的快照**（最旧能差 5 分钟）——拿不到实时额度。这两条直接违反"每次任务分配前都要重新查 `get_budget`、绝不依赖记忆中的额度数字"的纪律。

**A — fresh-if-stale：** `get_budget` 缓存在 `budgetFreshTtlSec`（默认 25 秒）内直接返回缓存，超过就向 daemon 请求一次**只读刷新**（`refreshSnapshotReadonly`：单次拉取 + 计算，**不动状态机**——读额度永远不会移动闸门），带 `requestId` 防慢回包错误地结算后来的等待者，single-flight 合并并发请求，超时/断连 fail-open 回退缓存。

**B — activity-gated idle advice：** daemon 记录最近活动时间戳，只有 balance / 欠载这类**建议**在 pair 空闲超过窗口时被抑制（活动恢复后指纹重置、再发一次）；**暂停 / 接力 / 恢复 / admission 这些关键信号永不被门控**。

为什么这样分——读额度是高频的决策前置动作，必须实时但绝不能有副作用（fail-open + 只读）；而建议是降噪对象，空闲时没人看、纯属噪音，但安全相关的状态变更一个都不能漏。

## 踩的坑 / 教训

- **探针限流 ≠ 额度耗尽。** provider 429/rate-limit 只是探针暂时刷不到新数据，**不触发任何干预**（不停机、不写 pending）——此时改用缓存 util 判阈值。把 429 误当额度耗尽去 checkpoint/停下是早期的错误反应。
- **探针数据质量分层是一切决策的前置。** 只有 `isDecisionGrade`（存在 fresh 且未过期的窗口）的数据才能驱动决策；不达标的只能展示。maximize **不豁免**这条——`resetEpoch` 未知、stale、燃尽率 `confident=false` 时，全部退回保守判据。这是 phantom-hold 不变量 I2 的来源。
- **单点尖峰会把悲观判定放大成误暂停**（Codex 对抗审 SUSPECT-5）：燃尽率的悲观侧取值要求"最近 3 样本中至少 2 点连续同向"才参与，等效 winsorize，否则一次突发就误停。
- **多 pair 同账号会集体怂恿过度并行**（SUSPECT-6）：每个 pair 各自发加速建议会叠加 → 加了**账号级 advice cooldown**（30 分钟内同账号同向建议不重复，且持久化到磁盘、不靠单 coordinator 内存）。
- **续接幂等墓碑绝不能随 kill 删**：`consumed/` 删了会导致升级/重启前已消费的续接被重新注入——这是设计时显式权衡过的"kill 有意不清理"。
- **resume sentinel 要 fail-open**：陈旧门 `degradedAt` 缺失/非数字时当作新鲜照常弹通知，绝不静默吞掉一个可能真实的恢复。
- **北京时区**：所有面向用户的额度时间（窗口重置、runway、reset_epoch）一律渲染成 UTC+8（#182）——用户在北京，看 `16:00Z` 要自己心算，体验差。

## 产出

- 一套完整的**额度智能**：时间感知动态暂停线（逼近 98% 用满又不被腰斩）、三态 admission 闸门（拒新任务但放收尾）、runway 跨套餐分配判据、烧不满加速建议——把"充分用满订阅额度"和"任务不被打断"这对矛盾真正调和了。
- 一套**全自动续接能力**：被硬线停在干净点后，窗口刷新即自动在原 TUI 续上未完成任务，Claude/Codex 双侧都是 fully automatic，幂等/重试/降级/GC 全链路硬化。
- 配套的**降噪与实时性**：fresh-if-stale 保证决策时拿到实时额度，activity-gating 保证空闲不刷屏，关键状态信号永不被吞。
- 全程跨引擎 cross-review：v3 设计稿经 Codex 对抗审落定 4 REAL + 2 SUSPECT + 2 RECOMMEND；P3/P4/#184 每个都经 Codex sign-off + 多轮 Claude review 收敛到 0 真实 issue；每个 PR 都配了真机 E2E 测试计划（额度协调 / P3 M3b admission / round-2/3 回归 / PR5 Codex idle 注入）。
