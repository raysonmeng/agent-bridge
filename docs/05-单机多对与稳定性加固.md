# 05. 单机多对 + 稳定性大加固（2026-05-20 ~ 06-10）

## 一句话定位

这一阶段，AgentBridge 从"一台机器只能跑一对、且单对偶尔会假死/即起即死"的**能用**状态，长成了"一机多对并行协作、且经过 12 轮交叉评审锤过的工业级稳定底座"——**从能用到耐用**的分水岭。

---

## 一、背景：单对稳了，但欠债浮出水面

到 05-20 前后，单对 Claude↔Codex 的协作链路已经跑通：MCP 推送、消息过滤、daemon 持久化、重连退避都在位。但真实的高强度使用把两类问题逼了出来。

**第一类是能力天花板。** 当时每台机器只能跑**一对**：单个 daemon，钉死在 `4500 / 4501 / 4502` 三个端口。可现实里我经常想同时开两份协作——一份在 worktree 里写功能、一份在另一个项目里做 review。第二份一启动，端口立刻冲突，状态目录互相覆盖，两对谁都别想干净地跑。"多开"从一个奢望变成了刚需。

**第二类是稳定性欠债。** 大量真实会话暴露了一串隐蔽的失败模式，最典型的是 **Issue #68 的"假死前端"**：Claude Code 进程被硬杀（崩溃 / `kill -9`）时，操作系统**不会发 FIN**，daemon 侧那条 WebSocket 就永远停在 `readyState=OPEN`。daemon 只凭 `readyState` 判断"槽位是否被占"，于是把新来的 Claude 会话一律以 `code=4001 "another Claude session is already connected"` 拒绝——日志里实测从 conn `#2` 一路拒到 `#19`，**连续 13 次、长达一个多小时连不上**。除此之外还有端口漂移复用导致的"即起即死"、孤儿进程残留、消息在背压下丢失、kill/doctor 输出难读等一系列"用着用着就硌一下"的问题。

单对能用，不等于多对耐用。这一阶段要同时补上**横向扩展（多对）**和**纵向硬化（稳定性）**两条线。

---

## 二、做了什么

### 1. 单机多对：路线 A（Shared-Nothing 多实例）

核心命题：**每一对 = 一个隔离的 daemon + 它自己的端口三元组 + 它自己的状态目录**，彼此 shared-nothing，互不知道对方存在。

- 每对占据一个 **slot**，slot 决定端口三元组，规律是第 N 个 slot 在经典基址上 `+ N*10`：slot 0 = `4500/4501/4502`（即经典端口），slot 1 = `4510/4511/4512`，以此类推。**slot 0 与升级前完全一致，所以单对用户 100% 不受影响。**
- 每对独立的 `daemon.pid` / `status.json` / 日志 / `killed` sentinel，统一收纳在 `<stateDir>/pairs/<pairId>/` 下。
- 用法上：`abg claude --pair work` + `abg codex --pair work` 配成一对；换个名字 `--pair review` 自动拿下一个空闲 slot；不带 `--pair` 时**由当前目录自动推导对身份**（realpath + 短 hash），同一目录里再跑就重连同一对。`abg pairs` 列出所有对，`abg kill --pair work` 只停一对、`abg kill` 停全部。

**最漂亮的一点是它的实现接缝：靠环境变量注入，而非核心代码手术。** 关键洞察是——**Claude Code 会把继承到的环境变量原样透传给它的 plugin MCP server**。所以"跑哪一对"纯粹靠在启动前设好 `AGENTBRIDGE_STATE_DIR` / `AGENTBRIDGE_CONTROL_PORT` / `CODEX_WS_PORT` / `CODEX_PROXY_PORT` / `AGENTBRIDGE_BASE_DIR` / `AGENTBRIDGE_PAIR_ID` 这几个变量即可完成，**`.mcp.json` 一个字都不用改**。每条 CLI 命令最顶端先跑一个"对解析器（pair resolver）"：在跨进程锁保护下到注册表 `<base>/pairs/registry.json` 分配/查找 slot，设好上述 env，然后既有的 `claude` / `codex` / `kill` 代码和 daemon——**因为本来就从 env 读这些值**——无改动地"自动工作"。

### 2. Stale frontend 探针驱逐（#68 / #73）

针对上面那个"假死前端"，修法是 **challenge-on-contest（争用时挑战）**："last writer wins"：

- 当第二个 frontend 在槽位"被占"时到来，daemon **不再盲信 `readyState`**，而是给在位连接发一个 liveness probe（ping），最多等 **3 秒**。
- 在位方按时 pong → 说明它真活着 → 用 `4001` 拒掉新来者（**保护活会话不被抢**，这是回归护栏）。
- 超时无 pong → 判定为半开死连接 → 用 `4002`（`CLOSE_CODE_EVICTED_STALE`）**驱逐**旧连接、**接纳**新来者。新会话在 `~3-5s` 内就能 ready，告别"重启 Claude 连不上一小时"的死循环。
- 并发争用串行化：两个新会话同时来抢一个死槽，**恰好一个赢**，另一个看到 `another liveness probe already in flight` 后可重试，杜绝 double-attach。
- 这套探针**驱逐的是探测意义上的"死"，不是 TTL 超时**——只针对确证无响应的连接动手。

### 3. 孤儿进程治理 + Codex turn 错误可观测性 + 跨 pair 自诊断

- **孤儿治理**：daemon / 子进程异常退出后残留的进程，不再无主漂着，引入显式回收与 `pairs prune` 清理路径。
- **turn 错误可观测**：Codex 一侧 turn 出错时，通过 `system_turn_aborted` 把状态显式吐给 Claude，不再让用户对着"卡住"干瞪眼。
- **跨 pair 自诊断**：多对并行后，配对错连、端口张冠李戴这类问题需要能被自检出来，doctor / 提示链路按 pair 作用域感知化。

### 4. P0a–P5 reliability hardening：12 轮 cross-review 的稳定性大加固（#101）

这是本阶段稳定性线的主体，**整整 12 轮交叉评审**（每轮全新独立 reviewer，避免 confirmation bias）才收敛到连续两轮 0 真实 issue。三大主题：

- **drift 复用（即起即死根因）**：端口/状态在漂移后被错误复用，导致 daemon"刚起来就死"。这是最隐蔽的一类——表面像启动失败，实则是复用了不该复用的漂移状态。
- **消息不丢失**：背压（backpressure）场景下消息曾会被丢弃，加固为不丢消息的投递。
- **kill / doctor 可读化**：把这两个高频运维命令的输出改成人类可读，降低排障摩擦；同时 EPIPE 日志硬化（#96）。

### 5. 删除可配置 pull 投递模式（#98，破坏性变更）

历史上消息投递有 push（channel 推送）和 pull（工具拉取）双模式可配。这一阶段**砍掉了 pull 模式**：投递统一为推送，失败时回落到由 `get_messages` 排空的内存队列。`AGENTBRIDGE_MODE=pull` 环境变量从此被忽略（仅一次性告警）。少一个模式分支，就少一类状态机歧义和测试矩阵。

### 6. install:global 一条命令 + dev 仓库守卫（#99）

`bun run install:global` 一条命令搞定"重建 CLI + 替换全局安装 + 同步插件 bundle"三件事（`--skip-plugin` 可选退出）；`dev` 子命令加仓库守卫，避免在非仓库 checkout 里误操作。解决的是"改完 `src/` 忘了重建插件 bundle、结果在测旧代码"这个反复踩的坑。

### 7. 发布链 P0 五连修 + 平台感知 + 探针数据质量

- **发布链 P0 五连修**：打包链路上的一串发布期炸弹（如 `build:cli` 不产 `dist/daemon.js`、daemon entry 默认值缺失等），集中修掉。
- **Windows 预检端口清理平台感知化**：端口 precheck 的 LISTEN filter 此前在 Linux CI 上超时（05-20 已先修），并把清理逻辑做成平台感知。
- **探针数据质量（#100）** + **`abg budget` 只读化（audit P0）**：保证额度探针读到的数据质量，并把 budget 命令收敛为只读。

> 注：预算协调层（#97）虽也在 06-10 这一批提交里起步，但它属于"额度协调"这条主线——**详见阶段 07**，本阶段不展开。

## 三、关键设计决策与为什么

### 决策一：多对为何选路线 A（多实例 Shared-Nothing），而不是路线 B（单 daemon 多 router）

这是本阶段最核心的架构取舍，也是被**主动废弃的 PR #81（路线 B）**反衬出来的。

- **路线 B（单 daemon 多 router）**：让所有对**共享同一个 daemon**，daemon 内部用多个 router 区分不同的对。听起来省进程、省端口。
- **致命缺陷——生命周期状态泄漏**：一旦多对挤在一个 daemon 里，**一对的 `kill` / `restart` 就会污染另一对的生命周期状态**。你想停掉 `work` 对，却可能连带搅乱正在跑的 `review` 对；一对崩了，可能把另一对一起拖下水。共享进程意味着共享故障域，生命周期再也无法干净隔离。
- **路线 A（多实例 Shared-Nothing）的取舍**：每对一个独立 daemon、独立端口、独立状态目录，**从物理上根除了这一类 bug**。代价是多开几个进程、多占几组端口——但进程和端口是廉价的，而"一对的操作绝不波及另一对"这个隔离保证是无价的。Shared-nothing 把"互不干扰"做成了**结构性事实**，而不是靠 daemon 内部小心翼翼的状态管理去"努力维持"。

一句话：**B 把隔离当成需要努力维持的运行时不变式，A 把隔离变成进程边界本身的物理事实。** 对一个要长期无人值守跑的本地桥来说，后者远更可靠。

### 决策二：跨进程锁为何是命门

路线 A 的 shared-nothing 看似简单，但它有**唯一一个真正的命门——slot 分配的跨进程锁**。

两个 `abg claude` 几乎同时启动时，如果 slot 分配不是原子的，两对就可能**抢到同一个 slot / 同一组端口**，shared-nothing 的隔离当场破产。这把锁是整个多对方案能不能成立的唯一咽喉，所以它被反复打磨（交叉评审里**迭代了 3 次**）：

- 用 `link(2)` 实现**原子锁文件**：先写临时文件，再 `link` 到锁路径，`EEXIST` 即判定"已被持有"。锁内含 owner pid + nonce。
- **陈旧锁回收只认进程存活、不认 TTL**：绝不抢占活进程的锁（否则它恢复后写回旧 registry 会造成 lost update）；回收动作本身还要再经一道独立的 serialized reclaim lock + 重新确认 owner 已死后才删，杜绝并发 evict 把活锁误删。
- 端口还会被**探测**：目标端口若被外部进程占用，直接报清晰的 `PAIR_PORTS_BUSY`，绝不悄悄连到错误的端口。

锁测试压到 **8 进程并发 + seeded-dead-lock，20+ 次 stress 0 碰撞** 才算过。**多对方案的全部正确性，最后都坍缩到这把锁上。**

## 四、踩的坑 / 教训

- **"只看 `readyState` 判断连接是否活着"是个陷阱。** Issue #68 的根因极其反直觉：TCP 连接的 `OPEN` 状态**不代表对端进程还活着**——硬杀进程时 OS 不发 FIN，半开连接能假死一个多小时。教训是：**生命周期判定必须用主动探针（liveness probe），不能信任被动的连接状态。** 这也是为什么修法刻意选"按存活探测、不按 TTL"。
- **真机复现是验证假死的唯一可靠手段。** #68 的核心测试只能靠 `kill -9 <pid>` 硬杀 Claude Code 进程、**绕过它正常关闭 WebSocket 的路径**来复现半开死连接——这种 OS 层面的半开态，单测的 fake clock 模拟不出真实感，必须真机手动跑。（自动化侧用 `liveness-probe.test.ts` 的 7 场景纯状态机覆盖逻辑，真机手动跑覆盖 OS 行为，两者互补。）
- **端口/状态的"漂移复用"是即起即死的隐形元凶。** 表面看 daemon 启动失败，实际是复用了不该复用的漂移端口/状态。这类 bug 不交叉评审基本挖不出来——它需要把整条启动时序在脑子里跑一遍。
- **"测试通过 + 类型检查通过"不等于 ship-ready。** P0a–P5 这一轮，正是用 12 轮交叉评审才从"看起来绿了"挖到"真的稳了"，捕获并修掉了一批跨进程锁竞争、env 双重语义、`kill --help` 误落 kill-all 等真实 bug。

## 五、产出

- **多对能力**：一机可并行跑任意多对 Claude↔Codex，端口 / 状态 / 日志完全隔离，单对用户零感知（slot 0 = 经典端口）。真实双对 E2E 验证通过：`kill --pair a` 只停 a、b 仍 LISTEN，`kill` 后端口全清。
- **一轮工业级稳定性底座**：假死前端探针驱逐、孤儿治理、turn 错误可观测、drift 复用根治、消息不丢失、kill/doctor 可读化、发布链 P0 全修、平台感知端口清理——经 12 轮交叉评审锤实。
- **工程纪律的固化**：`install:global` 一条命令收敛安装流程、删除 pull 模式收敛状态机、`budget` 只读化收敛副作用——少即是稳。
