# 04. 可靠性修复与故障复盘（2026-04-14 ~ 04-24）

## 一句话定位

这是 AgentBridge 从「功能堆叠」转向「被真实故障磨出韧性」的转折阶段——前三个阶段把双向协作的管道搭起来了，而这十天，项目第一次在真实用户的长任务现场被反复打脸：连接会假死、Codex 会静默卡住、resume 起不来、上游悄悄断线后 TUI 会闪退。每一次崩溃都逼出一条根因链，可靠性不是设计出来的，是从这些坑里一寸一寸长出来的。

## 背景：管道搭好之后，真实使用开始暴露脆弱点

前面阶段验证的是「Claude 和 Codex 能不能通起来」——握手、转发、消息过滤、双向协调，跑通了。但跑通和跑得稳是两回事。

进入 4 月中旬，桥开始被真实地长时间使用（用户在 `~/repo/quilin-agent` 里挂着跑长任务），暴露出一类此前从没认真对待的问题：**连接的生命周期管理**。之前的实现隐含一个乐观假设——连接要么活着、要么干净地关闭。现实里有大量「中间态」：进程崩了但 socket 还 OPEN、模型后端断流但 turn 事件不来、上游悄悄重连但会话状态没跟上。这些中间态恰恰是把人锁死、把进程搞挂的元凶。

这个阶段做的事，本质上都是在补「连接保活 / 重连 / 状态恢复」这三件长期被忽视的脆弱点。

## 做了什么

这十天的提交可以分成两条线。

**一条是工程治理与体验线**（4-14 ~ 4-15、4-21）：
- **延迟重放审批请求**——把 server-request 的审批请求延后到 `thread/resume` 完成之后再重放，避免在会话还没就绪时就抛出去。
- **auto 模式默认改 pull**——这是一个务实的退让：push 模式在某些情况下会静默丢消息，与其丢消息，不如默认走 tool pull，宁可慢也不丢（#21）。
- **配置契约清理 + 诊断日志**——把配置读写的契约理顺，并补上一批诊断日志（#40、#64）。这批日志看着不起眼，但下面要讲的第二次故障，正是靠它「第一次现场复现就抓到根因」。
- **日志路径迁移到 StateDirResolver**——日志统一收口到平台 state 目录。
- **npm postinstall 自动注册 marketplace + 装插件**——降低安装门槛。
- **abg init 注入多智能体协作段落**——把协作说明以 marker 方式幂等写进 `CLAUDE.md` / `AGENTS.md`，并增强 Codex 侧的协作 kickoff 消息，让两个 agent 一开局就知道彼此存在、知道怎么分工（4-15；4-21 修了其中两个 review bug）。

**另一条是可靠性硬骨头线**（4-24）：
- **非主动重连后重放 initialize**——上游 app-server 在非用户触发的情况下断线重连后，主动重放 `initialize`，把会话状态恢复回来，避免 TUI 在一个「全新但未初始化」的会话上继续工作。这条修复，直接对应下面第二次故障的根因。

## 重点：两次真实故障的根因与教训

这阶段最有价值的不是代码，是两份**真实故障的根因复盘**。它们示范了一种纪律：不靠猜、靠现场日志的字段证据把根因钉死，并辨析清楚「这个修复到底防的是哪个 failure mode」。

### 故障一：Codex 卡死 + resume 起不来（2026-04-18）

现场：用户长任务里 Codex 静默 1 小时以上，按 Esc 后 TUI 猝死，随后反复重启 Claude Code 全被 daemon 拒绝（`code=4001 another Claude session is already connected`），最后连 `abg codex resume` 也起不来。一个现象，扒出**四条独立根因链**：

- **A（P0）stale frontend 无法被驱逐**——daemon 判断旧 frontend 是否还活着，**只看 `ws.readyState`**。Claude Code 进程异常退出时 **OS 没发 FIN**，socket 残留在 OPEN 状态，daemon 又没有应用层 keepalive ping / idle timeout，于是老连接永远回收不掉，新实例连试 13 次全被拒。
  - 教训：**`readyState` 不等于「连接还活着」**。要么加 ping/pong 保活（推荐 15s ping、30s 未 pong 即 `terminate`），要么新连接到达时对老连接发 liveness challenge、超时即驱逐（last-writer-wins）。判活必须有应用层探针，不能信传输层状态。

- **B（P1）turn watchdog 缺失，Codex 静默把 Claude 永久锁死**——`turnInProgress` 只在收到 `agentTurnComplete` 事件时复位。那次现场，Codex 连发 3 条「message completed」却**始终没发 `agentTurnComplete`**，conn 上整整 **1 小时 21 分钟**零流量（模型后端断流或 app-server 卡住），于是 `turnInProgress` 永远 `true`，Claude 的 reply 三次被「Codex turn is in progress」拒绝。
  - 教训：**任何依赖「终止事件」复位的状态机，都必须配一个兜底 watchdog**——`lastEventAt + N 分钟静默` 强制复位，并发 `system_turn_completed_forced` 让用户知道这是兜底、不是真完成。N 的取值是**经验阈值**：推荐 3–5 分钟，因为日志里正常长推理一轮能到 15 分钟，太短会误杀正常 turn。

- **C（P2）activeThreadId 被响应字段覆盖**——`CodexAdapter` 直接用响应里的 `thread.id` 更新 `activeThreadId`。TUI 重连时 conn #2 resume 老 thread、conn #3 又 start 新 thread，两路响应交错到达，`activeThreadId` 被反复横跳，pending map 又以 `threadId:reqId` 为 key，thread 一切换旧响应就 `pending keys=[]` 失配——这正是「resume 后 thread 再也恢复不回来」的直接原因。
  - 教训：**状态变更必须由「显式请求」驱动，不能被任意到达的响应反向覆盖**；partition key 不能选一个会中途变化的字段。

- **D（P1）`abg codex resume` 起不来**——wrapper 对所有子命令用同一套 args 拼接，把 `--enable tui_app_server --remote ...` 拼在 `resume` 之前。但 clap 里这两个 flag 在顶层和 resume 子命令下**各自独立定义、并非 `global`**，于是它们被顶层命令吃掉，resume 子命令自己的 flag 是 `None`，结果 resume 退化成「在没有 bridge 的环境里 resume」，起了个跟 Claude 完全不通的孤立 TUI。
  - 教训：**包别人的 CLI 时不能假设 flag 是全局的**——要识别 TUI-mode 子命令（`resume` / `fork`），把 owned flag 插到子命令**之后**；同时别给 `exec` / `review` / `login` 这些本不该注入的子命令乱加 flag。

### 故障二：静默重连后「Not initialized」闪退（2026-04-24）

这次是上一阶段补的诊断日志（`f8698b8`）**第一次现场复现就立功**——抓到一条之前三轮 PTY 实验都没覆盖的全新根因。

现场三字段证据钉死了 failure mode：上游 app-server 在 `11:33:34` **自己掉线**（`intentional=false`、`turnInProgress=true`），1 秒后桥重连成功；**但 TUI 对此毫不知情**，在那个早已变成全新会话的连接上又工作了 3 分钟；直到用户按 Esc 触发 `turn/steer`，新会话回 `Not initialized`，TUI 从 `main()` 返 `Err`、exit(1)（runtime 36 分钟、`code=1`、无 signal）。

根因在自己代码里：**非主动重连路径（`handleAppServerClose` → `scheduleReconnect` → onopen 直接挂上新 socket）没有 replay initialize**；而主动重连路径（TUI 发 `initialize` 触发的 `reconnectAppServerForNewSession`）是有 replay 的。两条路径不对称，新会话是未初始化的，任何需要 initialized 状态的请求都被拒，TUI 视为 fatal。

这份复盘最精彩的，是它讲清了**为什么 phase 1 的 `outageQueue` 救不了**：
- `outageQueue` 只 buffer「断线期间 TUI 发出的消息」。但这个 case 里，断线只持续 1 秒、TUI 在这 1 秒内没发任何消息，queue 是空的；等 3 分钟后 TUI 发 `turn/steer` 时 socket 早已 OPEN，直接走 forward，撞上 `Not initialized`。
- 一句话点破：**`outageQueue` 防的是「消息丢」，不防「会话状态丢」——这是两个完全不同的 failure mode。**

教训沉淀为三条可复用的纪律：
1. **「重连成功」不等于「会话恢复」**——传输层重连之上，必须显式重放协议层的初始化握手（缓存 `initialize` / `initialized` 原文 + `threadId`，重连成功后按序 replay，任一步失败就降级关 TUI 1011 让用户感知，而不是让 TUI 在 stale session 上裸奔）。
2. **修复要分清防的是哪个 failure mode**——`outageQueue` 和 replay-initialize 看着都在「处理重连」，实则针对不同失效，缺一不可。把它们辨析清楚，才不会用错药、误以为已经修好了。
3. **诊断基建是可靠性的前提**——这次能「第一次复现就抓到根因」，全靠上一阶段补的 wrapper + adapter 三字段日志（`intentional` / `turnInProgress` / `appServerOpen`）。复盘还顺手把退出分类 regex 扩了 `not_initialized_after_reconnect` / `rpc_error_exit`，让以后同类闪退在 wrapper log 的 classification 字段里一眼可辨。

这份复盘同时体现了双引擎协作的工程文化：Claude 做现场诊断、定根因、出修法（含 A/B/C 三套方案与降级路径），并列出 4 个必须由 Codex 从 codex-rs 源码确认的风险点（`initialize` 是否幂等、`thread/resume` 能否在 fresh session 直接用、params 里有无 session-unique 字段、有无其他 bootstrap 请求）——**结论先交叉验证再动工**，而不是单方面拍板。

## 产出

- **更韧的连接判活**：从「信 `readyState`」转向应用层保活 / liveness challenge，根治「重启 Claude 连不上」。
- **更韧的 turn 生命周期**：watchdog 兜底复位 + 强制完成通知，Codex 静默不再把 Claude 永久锁死。
- **更韧的状态恢复**：非主动重连后重放 initialize，TUI 不再在 stale session 上闪退；`activeThreadId` 改由显式请求驱动，resume 风暴时状态不乱。
- **更顺的协作初始化体验**：`abg init` 幂等注入协作段落 + 增强 Codex kickoff，两个 agent 一开局就进入协作语境；`abg codex resume` / `fork` 参数位置修正，resume 能真正连回桥。
- **一套可复盘的诊断基建**：三字段时序日志 + 退出分类 regex，让下一次故障「全程 on record」。
