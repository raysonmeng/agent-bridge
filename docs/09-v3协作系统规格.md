## 1. 背景与目标

### 1.1 真实问题（两类场景，均需支持）

**A. 多人多仓协作（主场景）。** 多个开发者各自用自己的 coding agent（Claude Code / Codex / OpenCode / Cursor / Kilo……，CLI 与 GUI 混用、一人可能同时开多个异构 agent），**跨多个仓**并行开发，工作间存在**有向依赖**。痛点：各人的 agent 互相不知道对方在干嘛，导致**重复实现、冲突、浪费**。人与需求是多对多交叉关系。

**B. 单人多机，配额/算力池化（增量场景）。** 同一个人拥有多台机器 / 多个订阅账号（如 Mac + Linux 服务器，或公司机 + 家用机），各自有独立的 5h/周订阅额度与不同环境能力（GPU、内网访问、sandbox 执行）。痛点：单机额度用尽就卡住，或某些活只能在特定环境跑，而其他机器的额度/能力闲置。希望把多机的 agent 组成一个池，按**配额余量**与**环境能力**把工作单元路由过去。

> A 与 B 共用同一套底座（broker + room + 身份 + 事件总线 + adapter），区别在上层语义：A 是**协调**（知道谁干完了什么），B 是**调度**（把工作单元派给谁）。B 的调度层建在 A 的底座之上，见 §9。

### 1.2 目标

把 AgentBridge 扩展为**事件驱动的多人多仓 agent 协作系统**，并预留配额池化能力：
- **A**：一个人的 agent 完成一个工作单元后，自动把结构化完成事件扇出给同一 room 内相关成员的 agent，让每个人的 agent 在下一步行动时已知"谁干完了什么"。用 pub/sub（O(N)）替代轮询（O(N²)）。
- **B**：基于 presence + 配额余量，把可迁移的工作单元路由到合适的机器/agent（§9）。

### 1.3 核心设计原则

- **Edge + Backbone**
  - **Edge（每人本地，每 agent 一个薄 adapter）**：知道怎么把事件推进那个人当下正在跑的**任意异构 agent** 的活 session，以及检测其完成并发布。每个 agent 的适配活儿都在这层。
  - **Backbone（自带电池的单体 broker）**：默认一个自托管进程 + SQLite + token 认证，**零外部依赖即可跑**；所有外部基础设施藏在三个可替换接口后面，规模上来时按接口替换，core 不动。
  - 不犯两头的错：既不强迫小团队上 Kafka/SSO，也不在大公司已有基础设施时重造一个更差的。**接口 + 合理默认。**
- **控制面 / 数据面分离（★新增，见 §2.6）**：AgentBridge = 控制面（消息/事件）。**数据面（代码）= git。** broker 永不传文件。
- **可达性是"一台机器"级要求，不是"一套基础设施"**：broker 需要一个所有参与者可达的地址；Tailscale 让这台机器即使在内网/NAT 后也可达（§7）。

## 2. 核心概念模型

### 2.1 三层身份（person / logical agent / session）

身份分三层，职责不同，**不要混用**（注：v2 架构定义了 logical agent + session 两层，本规格在其上**新增 person 层**以支持多人）：

- **Person**：人，最粗。一人多需求会把事件混在一起，不能拿来绑 room。
- **Logical agent**：逻辑身份（如 "Alice 的 Claude Code"），durable。但它今天做 A 明天做 B，不能永久绑死某个 room。
- **Session**：当前活着的进程，ephemeral，重启换 id。**不能拿 session id 当 membership 的持久键**（一重启就掉出房间）。

事件署名（`from`）**精确到 logical agent id**（于是"Alice 的 Cursor 完成了 X"和"Alice 的 Claude"不混），需要时能上卷到 person。

### 2.2 身份唯一性：id 与 display name 分离

**显示名可以重复，唯一 id 绝不能重复。** 两个 Bob、一人多设备、一人多 agent 全靠这个区分。

- **identity id**：全局唯一、机器生成、内部主键。系统内部一切路由 / 署名 / 成员关系**全用 id，从不用名字**。
- **display name**：纯展示用，允许重复（两个 `Bob` 无所谓，因为系统从不靠它判断）。
- 信封里 `from`/`to` 带 **id**，UI 显示 **name**（撞名时显示成 `Bob (a1b2)` 或带邮箱）。

**唯一性来源（MVP 与 OIDC 一致）**：用**邮箱或 GitHub 用户名**当 id——天然唯一，且 OIDC 返回的 subject（`sub`）也是这类唯一标识，换 provider 时形状一致、core 零改动。

- 别让用户自己挑 id；`abg auth login` 时由 broker 生成或要求提供天然唯一标识。
- identity 表 id 列加 `UNIQUE` 约束；同一邮箱重复注册**复用同一 id**（同一个人多设备），不新建。
- **core 认 identity id，不认 token、不认 name。**

### 2.3 Room（协作的基本单位）

- **Room = 一个需求 / 工作流**，跨仓也跨人。一人多需求 = 其 agent 属于多个 room（v2 支持 + default room）；多人一需求 = 多成员 room。
- **任何人都可创建 room，别人 join**：`abg room create checkout` → 返回 id；`abg room list` 发现；`abg join`。
- membership（谁是 room 成员）绑 logical agent / person，**持久化在 SQLite，重启不掉**。

### 2.4 Workspace → Room 绑定

绑定的逻辑键是 **workspace（工作目录）→ room**，session 只是"当前活着的挂载"。三种 join 方式，按优先级：

1. **cwd → room 本地映射表（自动，主力）**：SQLite 存"绝对路径 → room id"。agent 起 session，adapter 拿 cwd 查表自动 join。**普通目录也能自动**，不碰仓库内容（避免标记文件被 commit 的污染）。
2. **显式 join（兜底）**：`abg join <room>` 或 agent 调 MCP 工具 `join_room`。任何场景成立，零前提。
3. **git worktree**：用 `git worktree` 的人天然拥有多个干净的独立 cwd（一需求一 worktree、物理隔离），每个映射到对应 room id。

> **worktree 不是依赖**，只是"让一个人同时拥有多个干净 cwd"的便捷手段。单需求的人一个仓一个目录，映射一条记录即可。绝大多数人走"映射表自动"或"显式 join"。

### 2.5 Session 记账与恢复

AgentBridge 记一张极小的表 `(workspace, agentType) → lastSessionId`，session 启动/结束时更新。

- 下次同一 workspace 起同类 agent，adapter 读到上一个 session id，调**该 agent 自己的恢复命令**接回上下文（Claude `--resume <id>`、OpenCode `run -c <id>`、Cursor 各自机制）。
- **边界**：AgentBridge 只**存和给** session id；真正的恢复动作由各 adapter 调 agent 原生命令完成。"记账在 AgentBridge，恢复在 adapter+agent"——上下文存储是各 agent 私有且异构的，你只持有钥匙（session id），不持有记忆本身。
- presence 事件携带"**新 session / 延续 session**"这个事实，让房间其他成员知道某人是冷启动还是接着干（决定要不要补背景）。判断与恢复本身仍是对方 adapter 的事，AgentBridge 只透传。

### 2.6 控制面 vs 数据面（★新增 —— 整个系统的地基约束）

这是本系统最重要的边界，决定了"跨网协作"为什么可行、为什么不传文件。

- **控制面 = AgentBridge 负责。** 即"谁在干什么、谁干完了什么、房间现在什么状态"这类**结构化信号**：完成事件、presence、DM、@提及、白板。信号很轻（一条事件几百字节），跨网零压力。
- **数据面 = git 负责，AgentBridge 绝不碰。** 即真正的**代码内容**。每个参与者在**自己机器、自己的仓库副本/worktree**里干活；代码靠**共享 git remote**（GitHub / 内网 Gitea / 任意 git 服务）push/pull 同步。完成事件里的 `repo` / `branch` / `commit` 字段就是指向 git 的指针，而非文件本身。

**推论（务必内化）**：
1. broker 永不传输文件、永不远程读写他人磁盘。"让远程 agent 操作我本机文件"是明确越界（重、慢、不安全）。
2. 协作的前提是**参与者共享同一个 git remote**。这是用户侧的约定，不是 AgentBridge 提供的能力。
3. "新成员没有上下文"的解法是给他**房间协作上下文**（白板，§4），不是给他代码（代码他自己 git clone）也不是恢复他 agent 的私有记忆（做不到）。
4. B 场景（§9）派发工作单元时，被派方必须**已能通过 git 拿到对应代码**（已 clone 或现场 fetch/worktree）；派发的是"工作单元描述 + git 指针"，不是迁移半截 session。

## 3. 消息与事件模型

### 3.1 信封（复用 v2）

复用 v2 的 message envelope，关键字段：`roomId`、`from.agentId`、`to.agentIds`、`mentions`、`idempotencyKey`、`traceId`、`deliveryMode: online_only | store_if_offline`、`ack`。

### 3.2 三档路由

daemon 看信封路由，三档覆盖：

- **广播**：不填 `to` → fan-out 给 room 所有成员。
- **@提及**：填 `mentions` → 投房间，但只高亮被 @ 的人。
- **DM（私发）**：填 `to: [agentId]` → 只投目标，房间内其他 session **完全不经手、不感知**（避免无关打扰）。

定向投递同样保留 **loop prevention**（不回发给 `from`）与**离线补投**（目标不在线 → `store_if_offline`）。

> **防环升级（对齐跨机/多方）**：v1 的"按二元 source 取反"防环在多方/链式转发下失效。多方场景下，每条信封带 `traceId` + 可选 `hop` 计数，daemon 路由时记录"已投递集合"，超 hop 上限丢弃。这是从二元 source 到显式寻址防环的泛化。

adapter 暴露成 `abg send --to <agent>` 或 MCP 工具一个 `to` 参数。

### 3.3 完成事件（task_completed）

新增事件类型，结构化字段（见附录 A schema）：`summary`（一句话人类可读）、`repo`、`branch`、`commit?`、`contract?`、`unblocks?`（房间内相关性高亮用）。

- **发布侧**：复用 `plugins/agentbridge/hooks/hooks.json`，把 agent 完成钩子（Claude `TaskCompleted`/`Stop`、Cursor `stop`、OpenCode `session.idle`……）接到 `abg publish`，干完自动发进当前 room。再补 MCP 工具 `announce` 作为 agent 主动宣布的手动路径。
- **订阅侧**：复用现有 channel 推送（`claude-adapter.ts`）/ `turn/start` 注入（`codex-adapter.ts`），新增对该事件类型的分支。
- **完成的判断权（MVP）**：默认 agent 钩子触发 = **软通知**（"一轮结束"≠"真完成"，仅作进度）；可选由人确认 `abg done` 下硬判断。**不做 CI 背书那一档**（与跨 room 一起砍，见第 10 节）。
- **噪音控制（重要）**：多人多 agent 下每个 `Stop` 都发会刷屏。`abg publish` 内必须做**去重 + 节流**（如同一 `(agentId, repo, branch)` 短时间窗内合并、空内容轮次不发），附录 C 说明。

### 3.4 控制平面事件 / Presence

成员 join/leave 是控制事件，走同一推送通道但单独归类。复用 v2 的 `agent_online` / `agent_offline` / `room_updated`，纳入：

- 成员加入 → 向该 room 现有成员广播 `member_joined { roomId, agent:{id,name,type}, sessionContinuity:"new"|"resumed" }`，各 adapter 注入自己 session（"Bob 的 Cursor 加入了 checkout（新 session，无历史上下文）"）。
- leave 同理，含 session 异常断开（靠现有 inactivity watchdog 触发）。
- daemon 维护 room→members 内存视图（SQLite 持久化），状态变更即推。
- **presence 携带机器/能力标签（为 B 铺路，★新增）**：online 事件可带 `host`（机器标识）、`capabilities`（如 `sandbox-exec` / `gpu` / `intranet-db`）、`budgetHint`（配额余量摘要，来源见 §9.4）。A 场景忽略这些字段不影响；B 场景的调度依赖它们。

## 4. 房间记忆：底账 + 白板

两层分离，职责不冲突、缺一不可：**底账保真，白板保用。**

### 4.1 底账（append-only event ledger）

每条信封原样落盘，**不可变**，是"发生过什么"的事实真相。审计、回放、追溯靠它。**只存信封，不存 payload 大块**（不存 agent 完整输出，只存那句 `summary`）。

### 4.2 白板（living summary）

一份**被分析整理过、会被覆写**的房间当前状态摘要。不是事件堆叠，是事件的**蒸馏**。新成员 join 读白板（不是回放底账）即可知道"房间现在是什么状态"。

**谁来蒸馏 —— MVP 用结构化槽位机械合并，不上 LLM**：白板不是自由文本，而是几个固定槽位：

- `contracts_ready[]`、`in_progress[]`、`blockers[]`、`recent_milestones[]`

事件按类型更新对应槽位（完成事件 → 往 `contracts_ready` 加一项）。纯规则、零 LLM、零冲突、零成本、永不暴增。
> 不让多个异构 agent 并发写自由文本白板（会互相覆盖/打架）；允许 agent 主动贴一条到 `recent_milestones`（`abg note`）作为补充，但主力是机械合并。**LLM 蒸馏留作演进**，等结构化槽位不够表达再上。

### 4.3 大小控制（不会暴增）

**存事件，不存对话**——会变大的是逐字对话流，你根本不存那个。一条完成事件几百字节，一个房间做几天撑死几 MB。再加：

- 只存信封不存大块 payload；
- **滚动保留**：每 room 留最近 N 条（如 500）或最近 M 天，老的归档/删除；
- 白板每 room **一行**，覆写式，天然封顶。

### 4.4 新成员上下文

成员 join → 读白板那一行注入 session。这是"新成员没上下文"的**真正解法**：不是恢复他自己 agent 的记忆（做不到也不该做，那是 adapter 的事），而是给他**房间的协作上下文**（统一、很轻、你完全能给）。

### 4.5 SQLite 表

- `room_events`：append-only 底账。
- `room_whiteboard`：每 room 一行，JSON 存那几个槽位。事件到达 → 更新底账 + 按规则更新白板行。
- 可选命令：`abg whiteboard`（查看）、`abg note`（贴一条到 `recent_milestones`）。

## 5. Adapter 契约（Edge 层）

### 5.1 取舍

- **没钩子的 agent 不支持**（如 Zed、某些 IDE 模式）——强行代理只会拖垮维护。能做的逐个做。
- **不深度代理**。adapter = 薄薄一层，包在每个 agent **原生 hooks（发布）+ MCP client（接收）** 之上。Codex 现有的 app-server 代理是最脆弱的那种，仅此一家保留。

### 5.2 最小契约（每接一个 agent 只填这三件事，daemon 侧不变）

1. **register**：会话启动时向 daemon 注册身份 + 按 §2.4 解析并 join room（查 cwd→room 映射 / 读 worktree 标记 / 显式 join）。
2. **publish-on-completion**：挂到该 agent 的完成钩子，把结构化完成事件发进 active room。
3. **receive-into-session**：把收到的事件（完成事件 / DM / presence）推进该 agent 的活 session（MCP 推送通道或等价机制）。

### 5.3 难度分档（按团队实际在用的排优先级）

OpenCode / Kilo 最易（25+ 事件、插件模型）→ Claude Code 易（hooks+MCP 一等公民）→ Cursor 中（有 hooks 但事件语义要归一化）→ Codex 偏重（app-server 代理）。

### 5.4 一人多异构 agent

每个 agent 注册成各自的 logical agent id，归属到同一个 person。互不混淆。

## 6. 三个可替换接口与自带电池实现（Backbone 层）

### 6.1 接口定义（core 只依赖接口，永不直接依赖具体实现）

```typescript
interface MessageTransport {
  publish(topic: string, msg: Envelope): Promise<void>;
  subscribe(topic: string, handler: (msg: Envelope) => void): Unsubscribe;
}
interface Store { /* membership / rooms / cwd_room_map / session 记账 / room_events / room_whiteboard / pending */ }
interface IdentityProvider {
  authenticate(credential: string): Promise<Identity>;  // -> { id, displayName }
}
```

### 6.2 自带电池实现（零外部依赖，单机可跑）

| 接口 | 最简实现 | 说明 |
| --- | --- | --- |
| **MessageTransport** | 进程内事件总线 + WebSocket 扇出 | daemon 本就是消息路由器。所有 adapter 通过 WSS 连到这一个 broker 进程，进程内用内存 pub/sub（EventEmitter 级）路由，WS 直推订阅 session。无 Kafka/NATS。 |
| **Store** | SQLite（WAL 模式） | 单文件零运维。WAL = Write-Ahead Logging：写入先追加到独立 `-wal` 日志、读仍读主库，于是读写并发、写不阻塞读，且崩溃不坏库、写更快（`PRAGMA journal_mode=WAL`）。所有持久化数据都在这一个库。 |
| **IdentityProvider** | broker 签发的 pre-shared token（PSK） | PSK = 预共享密钥：`abg auth login` 颁一个绑唯一身份的 token，broker 比对放行。零依赖立刻能用；无集中管理，适合小团队。 |

### 6.3 为什么需要认证（不是抽象安全，是路由的基础数据）

认证在这套系统里是**路由和协作的基础数据**，几处离了身份就没法工作：

- **事件署名**：完成事件得标明"**谁**干完了"——没认证谁都能冒充发"契约已就绪"，污染房间状态。
- **DM 定向**：`to:[id]` 要解析到真实身份并只投给它。
- **room 成员关系**：谁能 join、谁能读白板/历史，都是对身份的判断；多团队还要隔离。
- **presence**："Bob 的 Cursor 加入了"——这个 Bob 来自认证后的身份。

所以哪怕最简版也得有（PSK 那种最朴素的即可）。注意：应用层身份（PSK/OIDC）与**网络层准入（Tailscale ACL，§7.4）是两层**，纵深叠加。

### 6.4 无缝替换原则（必须从第一天兑现）

- **接口先行 + 双实现并行**：哪怕生产版先放 skeleton。只做单实现以后再抽，几乎必然抽漏（接口被自带电池版细节污染）。
- **一套契约测试，两套实现都必须过**，保证可替换性是真的不是嘴上的。
- **别让实现细节渗进 core**：PSK 版必须产出和 OIDC 版**一样形状**的 `Identity`（都给 `{id, displayName}`）。core 认 identity id，不认 token/name。
- 换 backend = 改一行配置选 driver，core 零改动。

### 6.5 生产 backend 映射（接口后面的可选项，小团队一个都不碰）

| 关注点 | 生产 backend | 是什么 |
| --- | --- | --- |
| MessageTransport | **NATS** / Redis Streams / Kafka | 开源消息中间件。可集群部署做到无单点高可用。NATS 在同类里轻量好用；其 **leaf node** 可做"每机本地 + 选择性跨机转发"的联邦（跨地理/边缘场景）。 |
| Store | Postgres | 关系数据库，规模上来替 SQLite。 |
| IdentityProvider | **OIDC** / SAML + RBAC | 大公司 SSO 多是 OIDC/SAML，接进来复用员工身份。 |
| 可观测/审计 | 公司 SIEM | —— |
| 控制平面真相源 | 从公司 tracker 同步 | 见 §10：MVP 自带最简 room registry；外部 tracker 是**可选输入**，不是依赖。 |

## 7. 网络与可达性（★新增 —— Tailscale 集成）

> 这一节补全原 spec 仅一句"broker 必须有所有人可达的地址"的留白：给出**三种部署形态**，并把 Tailscale 作为"不买公网 VPS、用内网常开机器"时的首选可达方案。基于 2026-06 Tailscale 官方文档调研。

### 7.1 硬约束：broker 需要一个所有参与者可达的地址

跨机器扇出绕不开"有一台机器对所有 adapter 可达"。这是**一台机器级**要求，不是一套基础设施。broker 进程必须常开（开发机会休眠/关机，不适合兼任 broker）。

### 7.2 三种 broker 部署形态（按场景选）

| 形态 | broker 跑在哪 | 可达方式 | 适合 |
| --- | --- | --- | --- |
| **公网 VPS** | $5 VPS | 公网 IP/域名 + TLS（WSS）+ PSK | 成员分散在公网两端、不想碰 VPN |
| **内网常开机 + Tailscale（推荐默认）** | 团队里任一台常开机（台式机 / Mac mini / NAS） | tailnet `100.x` 直连 / MagicDNS 名 + ACL | 不想买/维护公网 VPS，成员愿装 Tailscale |
| **纯局域网** | 局域网常开机 | 局域网 IP（同子网才通） | 所有成员同一物理网络，最简 |

### 7.3 Tailscale 集成方式（直连，不用 serve）

- **绑定**：broker 的 WSS 监听从只绑 `127.0.0.1` 改为**可配 host**；跨网时绑 **tailscale0 接口的 `100.x` IP**（**不要绑 `0.0.0.0`**，那会同时暴露给物理 LAN/WiFi）。其他机器直连 `wss://100.x:<port>` 或 MagicDNS `wss://<broker-host>.<tailnet>.ts.net:<port>`。
- **不要用 `tailscale serve`**：官方文档未明写支持 WebSocket，社区实测有坑（query 参数被剥离、长连周期性断、版本回归）。直连 `100.x` 无反代层、延迟最低、行为可控。
- **MagicDNS 发现**：用机器名而非记 IP；`abg join <broker-host>.<tailnet>.ts.net --room <加入码>`。
- **长连韧性**：首连可能先走 DERP 中继再透明升级直连；路径切换对 TCP 透明，但 adapter 仍需 ws 心跳 + 断线指数退避重连（与 §8.2 的无感重连同一套）。

### 7.4 双层鉴权（网络层 Tailscale + 应用层 IdentityProvider）

两层纵深，缺一不可：

- **网络层（Tailscale ACL）**：用 grants 限定只有打了 `tag:agent` 的设备能访问 broker（`tag:broker`）的端口，例：`{src:["tag:agent"], dst:["tag:broker"], ip:["tcp:<port>"]}`。**务必先移除 tailnet 默认 allow-all**（"没写 acls 段 = 全通"是大坑，删掉不等于关闭）。模板见附录 E。
- **应用层（IdentityProvider，§6）**：PSK/OIDC 解析连入方的**协作身份**（identity id），用于署名/路由/成员判断。
- 服务端可选用 `tailscale whois <ip:port>` 校验连入设备的 tags（**别依赖 `Tailscale-User-*` header，对 tagged 设备为空**），作为网络层的额外信号。
- 关键原则：**不能假设"在 tailnet 内就可信"**。网络层挡住非授权设备，应用层确定"你是谁"。明文 ws 也要靠应用层 token；走 Tailscale 时链路本身已是 WireGuard 加密。

### 7.5 用户预装与入网

- 参与者预装**系统级 Tailscale（内核模式，非 userspace）**，登录同一 tailnet。AgentBridge（TS/Bun）**不嵌入任何 Tailscale 库**（`tsnet` 仅 Go；`tailscale-js` 是 Bun-only 实验项目，不可靠）——应用只 connect 普通 `100.x` IP，对 Tailscale 无感。
- 无人值守机器（如 broker、B 场景的算力机）用 **ephemeral + tagged auth key** 自动入网；auth key 泄露要主动 Revoke。

### 7.6 坑（落地必查）

1. broker 服务必须绑 `100.x`/可路由地址，**不能只绑 loopback**（绑 loopback 收不到经 tailscale0 进来的连接）。
2. tailnet 默认 ACL 是 allow-all，**必须主动收敛**。
3. 容器若跑在 userspace-networking 模式，需走 SOCKS5 代理（普通桌面/服务器内核模式不受此限）。
4. **禁用 Funnel**（公网暴露），内部控制通道只在 tailnet 内。
5. 首连走 DERP 再升级直连，加 ws 心跳 + 重连。

> **未验证项（落地前实测）**：直连命中率/延迟、长连稳定性。建议先做一个最小 spike（两台已组 tailnet 的机器，手动绑 `100.x` + 配 ACL，验证一条 WSS 长连稳定），再定稿。

## 8. 高可用与稳定性

### 8.1 诚实判断

**单体自带电池版 + 真正零抖动 HA 是冲突的。** 真正零抖动（无单点、滚动重启不丢、分区可用）需要复制/共识/故障转移——那就是 NATS 集群 / Redis Sentinel / Postgres 复制，即你想给小团队省掉的基础设施。务实做法是分清两类抖动。

### 8.2 A 类：进程内抖动（能也必须做到接近零）

即便单实例，下面这些让它稳如磐石：

- **客户端无感重连**：adapter 与 broker 断开时本地排队 + 指数退避自动重连，**重连期间业务 session 完全无感**（事件先攒着）。这是体感"零抖动"的最大来源，也是跨网（DERP↔直连切换、漫游）的必备。
- **持久化即送达**：事件先落 SQLite（WAL）再确认，进程崩溃重启后从 pending 续投，**不丢**；配合 `idempotencyKey` 去重，续投不重复打扰。
- **优雅重启**：升级时排空在途、新实例起来再切，systemd/supervisor 自动拉起。
- **健康自检 + watchdog**：现有 inactivity watchdog 扩成进程级健康检查 + 自动重启。

### 8.3 B 类：实例级故障（单实例消灭不了，只给路径）

broker 那台机器宕了，单实例必有不可用窗口。自带电池版做不到 B 类零抖动，但 `MessageTransport`/`Store` 接口设计成**允许多实例 + 外部存储**，让"需要真 HA 时切到 NATS 集群 + Postgres"这条路**从第一天就预留好**，core 不改。

### 8.4 定调

自带电池版目标 = **单实例、极强韧性、客户端无感重连、持久不丢、崩溃自愈 → 体感零抖动**；真正多副本零停机 HA 通过替换生产 backend 达成。承诺"单进程也无单点"是物理上做不到的。

## 9. B 场景：配额池化与工作单元路由（★新增 —— A 底座之上的调度层）

> B 建在 A 的全部底座（broker + room + 身份 + presence + 事件总线 + adapter）之上。**A 是协调，B 是调度。** B 是演进层，不在单团队 MVP 必做范围，但其依赖的 presence 字段（§3.4）应在 MVP 就预留。

### 9.1 B 与 A 的本质区别

- A（协调）：被动告知"谁干完了什么"。事件是**已发生**的事实。
- B（调度）：主动决定"这个活派给谁干"。要在多个候选 agent 里**选一个**执行。

### 9.2 数据面约束（B 最容易做空的地方）

承 §2.6：被派方执行工作单元，必须能拿到对应代码。所以：

- **派发的是"自包含工作单元"**：`{ repo, branch/baseCommit, 任务描述, 验收标准, 关联 room }`，**不是迁移半截 session / 不是传文件**。
- 被派方 adapter 收到后：① 确保本地有该 repo（已 clone 或现场 `git fetch` + worktree）；② 在干净 cwd 起一个 agent session 执行；③ 完成后照 §3.3 发 `task_completed`（commit 推回共享 remote）。
- **不可迁移的活别派**：依赖本机特定本地状态（未提交改动、本地未推送分支、特定本地数据）的活不进调度池。调度池里只放"换台机器也能干"的工作单元。

### 9.3 设计：presence + 配额 → 路由决策

- **候选集**：room 内在线、且 presence 声明具备所需 `capabilities`（如某活需 `sandbox-exec` 或 `gpu`）的 agent。
- **打分**：在候选集里按 `budgetHint`（配额余量，§9.4）+ 负载（当前是否 busy，复用 turn 状态）排序，选余量最高且空闲者。
- **协议（轻量，复用 envelope）**：
  - `task_offer`（发起方 → room 或定向）：广播一个工作单元待认领。
  - `task_claim`（候选方 → 发起方）：声明认领（先到或按打分指派）。
  - `task_assigned`（broker/发起方 → room）：确认归属，避免多方重复干（本质是 A 的"防重复"在调度方向的应用）。
- **MVP-B 范围**：先做**半自动**——`abg offer <工作单元>` 把活 offer 出去，broker 按余量给出**推荐**人选，由人确认或 agent 自动 claim。**全自动调度器留作演进**（自动判断可迁移性 + 自动指派风险高）。

### 9.4 与现有 quota guard 整合

- AgentBridge 已有 budget/quota guard（轮询两侧账户 5h/周窗口、`get_budget`、`system_budget_*` 通知）。B 复用它：每个 adapter 周期性把本机/本账号的余量摘要随 presence 上报为 `budgetHint`。
- broker 据此维护"room 内各 agent 余量视图"，供 §9.3 打分。
- 与现有"减速/暂停/续接"协议一致：余量见底的 agent 不进候选集（等同 pause 语义）；窗口刷新后重新进入。

### 9.5 明确边界（B 的 MVP）

- 只做**可迁移工作单元的半自动 offer/claim + 余量推荐**。
- **不做**：迁移运行中的 session 上下文、自动判定任意任务可迁移性、跨机实时同改同一文件。
- 数据面始终是 git；broker 永不传代码。

## 10. 明确不做（Out of scope / 已砍）

- **没钩子的 agent**（Zed 等）——不支持。
- **dependency edge / 跨 room 路由**——MVP 砍掉。需跨 workstream 协调时，先把相关的人拉进同一个多成员 room；真有规模痛点再回头做。
- **CI 背书的"完成"判断**——随跨 room 一起砍。MVP 完成判断 = agent 钩子软通知 + 可选 `abg done`。
- **恢复成员 agent 的私有记忆**——是各 adapter+agent 的事，AgentBridge 只存/给 session id。
- **传输代码文件 / 远程读写他人磁盘**——明确越界（§2.6）。数据面永远是 git。
- **硬依赖外部 tracker（Jira/Linear）**——自带最简 control plane（room registry），外部 tracker 作为**可选 pluggable 输入**。
- **LLM 白板蒸馏**——留作演进，MVP 用结构化槽位机械合并。
- **B 的全自动调度 / session 上下文迁移**——演进层，MVP-B 只做半自动 offer/claim（§9.5）。
- **嵌入 Tailscale 库 / `tailscale serve` 暴露 WebSocket**——不做；靠系统级 Tailscale + 直连 `100.x`（§7）。

## 11. 三档演进路线

### 11.1 单团队 MVP（当前目标，A 场景，跨机）

物理形态：**一个常开 broker 进程（内存 pub/sub + WSS）+ 一个 SQLite 文件（WAL）+ PSK token + 每 agent 一个薄 adapter**，跑在一台公网 VPS / 内网常开机（+ Tailscale 可达）上。外部基础设施需求 ≈ 零。包含：

- 三个可替换接口 + 自带电池实现（§6）
- 薄 adapter 契约：register / publish-on-completion / receive-into-session（§5）
- 三层身份 + id/name 分离 + 邮箱/GitHub 名做唯一 id + UNIQUE 约束（§2.1–2.2）
- Room（任何人可建、可 join）+ cwd→room 映射自动 join + 显式 join 兜底（§2.3–2.4）
- session 记账 + 新/延续标记（§2.5）
- 控制面/数据面分离落实：broker 不碰文件，文档化 git 作数据面的协作前提（§2.6）
- 信封三档路由：广播 / @提及 / DM + loop prevention（traceId/hop）+ 离线补投（§3.1–3.2）
- 完成事件（hook 发布 + `announce`）+ 去重节流 + 房间内相关性高亮（§3.3）
- presence 事件 member_joined/left + 预留 host/capabilities/budgetHint 字段（§3.4）
- 底账 + 白板（结构化槽位机械合并）+ 新成员摘要注入（§4）
- **网络层 MVP**：broker host 可配 + Tailscale 直连 `100.x` + ACL 模板 + 双层鉴权（§7）
- A 类韧性：落盘续投 + 无感重连 + 优雅重启 + watchdog（§8.2）

### 11.2 多团队演进 + B 场景

- 多团队隔离（room/topic 授权）；GitHub OAuth 升级（替换 PSK，走 `IdentityProvider`）
- **B 场景 MVP**：半自动 offer/claim + 余量推荐 + quota guard 整合（§9）
- 房间历史更丰富的检索 / LLM 白板蒸馏
- （重新评估）dependency edge 与跨 room 精准路由
- 选择性同步外部 tracker 作为 control plane 输入

### 11.3 生产 HA

- `MessageTransport` → NATS 集群（或 leaf node 联邦）/ Redis Streams；`Store` → Postgres；`IdentityProvider` → OIDC/SAML + RBAC
- 多实例横向扩展、零停机滚动升级（B 类抖动消除）
- B 的全自动调度器（含可迁移性判定）
- 可观测接公司 SIEM、限流、DLP、审计

## 12. MVP 数据模型汇总（一个 SQLite 库）

- `identities`：id(UNIQUE, 邮箱/GitHub 名) / display_name
- `agents`：agent_id / person_id / type（logical agent）
- `sessions`：session_id / agent_id / started_at（当前/历史 session）
- `workspace_sessions`：(workspace_path, agent_type) → last_session_id（session 记账）
- `rooms`：room_id / name / created_by

- `cwd_room_map`：workspace_path → room_id（自动 join）
- `room_events`：append-only 底账（结构化信封）
- `room_whiteboard`：room_id → 一行 JSON（contracts_ready / in_progress / blockers / recent_milestones）
- `pending_deliveries`：离线补投队列（含 idempotency_key）

- （B 演进）`task_offers`：offer_id / room_id / unit(JSON: repo,branch,desc,acceptance) / status / assigned_agent_id

## 13. MVP 端到端验收标准

> 在两台以上不同机器、跨多个仓的场景下，某开发者用任一受支持的 agent 完成一个工作单元后，**无需任何人手动同步、无任何轮询脚本**，同一 room 内相关成员的 agent 在其下一步行动时自动获知该完成事件（含一句话摘要、涉及仓/契约）；通过 DM 发给特定成员的消息不会打扰房间内其他 session；新成员 join 时能从白板拿到房间当前状态摘要；离线成员重连后补收到事件；同一 workspace 重起同类 agent 能接回上次 session 上下文；两个同名 Bob / 一人多设备 / 一人多 agent 在路由与署名上绝不混淆；broker 经 Tailscale `100.x` 在内网可达且非授权设备（无 `tag:agent`）连不上；**broker 全程不传输任何代码文件**；且原有 Claude↔Codex 单机对话流不受影响。逐项可断言通过即 MVP 完成。

> **B 场景验收（演进档）**：发起方 `abg offer` 一个自包含工作单元后，broker 依据 room 内 presence 的 capabilities + budgetHint 给出推荐人选；被派 agent 在另一台机器上 `git fetch` 到对应代码、起 session 执行、完成后 `task_completed` 推回 commit；配额见底的 agent 不被推荐；全程无文件传输、无 session 上下文迁移。

```typescript
    unblocks?: string[];      // 房间内相关性高亮（agentId 或 "topic:xxx"）
  };
  timestamp: number;
  deliveryMode: "store_if_offline" | "online_only";
  to?: string[];              // 填则为定向(DM)；不填为房间广播
  mentions?: string[];
  ack?: { requested: boolean };
}
```
