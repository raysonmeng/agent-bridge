# 14 · 非破坏部署（升级不杀活跃会话）

> 定位：把 backlog ⑥「`install:global` 现在要 `--force` 杀全部会话」从痛点落成方案，作为需求真相源 + 技术方案。
> Owner：aikey@rayson（独占文件：`scripts/install-global.mjs` / `scripts/install-safety.cjs` / `src/daemon-lifecycle.ts`）。**不触碰** broker.ts / room-bridge.ts / codex-adapter / room-* / collab-store（房主 ①② 域）。

## 1. 原始需求（What + Why）

**用户故事**：我和同事各自开着 `agentbridge claude` 在干活，这时想升级 AgentBridge 到新版。今天 `bun run install:global` 会：①检测到活跃会话就中止，要我加 `--force`；②`--force` 装完后跑 `stop-running` 把运行中的 daemon 全停掉 → **我俩正在进行的会话全断**。

**为什么是问题**：升级是高频运维动作，却和"别打断正在干活的会话"直接冲突。结果是大家要么不敢升级（版本漂移越拖越久），要么挑没人干活的时间升级（运维窗口受限）。违反 `interaction-simplicity-first` + 主链路鲁棒性。

**目标**：升级全局安装**不打断任何活跃会话**——老会话用老字节服务到自然结束，新字节在下一次会话启动时生效。`--force` 不再等于"杀全部会话"。

## 2. 现状分析（Why it's like this today）

`scripts/install-global.mjs`（local/npm 两路）当前流程：
1. **preflight**：扫 `ps` 找活跃 AgentBridge 前台 / Codex TUI；有活跃会话则要求确认或 `--force`（`install-global.mjs:14,61`）。
2. **build + verify first**：红构建在杀任何东西之前就中止（`:431` 注释），这点已经是对的。
3. **install**（`npm i -g --force` / 本地 tarball）。
4. **`stop-running`**（`install-safety.cjs stop-running`，`:447/:530`）：装完后停掉运行中的 daemon。

矛盾点就在第 4 步：daemon 是**按 pair 长驻**的（survives 跨会话），`stop-running` 一停，挂在它上面的活跃 `bridge.ts`（前台 MCP）连接全断 → 用户会话被打断。注释里"old daemon keeps serving until the new bytes are on disk"只覆盖了"装的过程中"不停，没覆盖"装完之后别停"。

**根因**：升级用了"停掉老 daemon 强制换代"的模型，而不是"版本共存、安全点切换"的模型。

## 3. 方案（版本共存 + 安全点自切换）

### 3.1 核心思路

不在升级时停 daemon。改为：
- **老 daemon 继续用老字节服务活跃会话**（进程已在内存里，磁盘换字节不影响它）。
- **新字节只对"下一个全新 daemon 进程"生效**。
- daemon 检测到磁盘字节比自己新（已有 build-drift 检测，见 `abg doctor`）时，**在安全边界自重启**：仅当本 pair 无活跃 turn（会话空闲 / 两 turn 之间）才换代，绝不打断进行中的 turn。

### 3.2 为什么这样设计（对比方案）

| 方案 | 升级即停 daemon（现状） | **版本共存 + 安全点自切换（选）** | 完全不切换（手动重启） |
|---|---|---|---|
| 活跃会话 | ❌ 被杀 | ✅ 不打断 | ✅ 不打断 |
| 新版生效 | 立即 | 下个 turn 边界 / 下次启动 | 要用户手动 kill 重启 |
| 实现成本 | 0（现状） | 中（加版本检测+安全点判断） | 低 |
| 漂移风险 | 无 | 低（自切换收敛） | 高（用户忘了重启就长期跑老版） |

选中间方案：既不打断会话（主链路鲁棒），又能自动收敛到新版（不留漂移），代价是一段"版本检测 + 空闲判断"逻辑。完全手动重启会留版本漂移（违反持续交付收敛性）。

### 3.3 行为细节

- `install:global` 默认**不再调 `stop-running`**；新增 `--restart-now`（旧 `--force` 语义的显式版）给"我就是要立刻全停换代"的场景。`--force` 退化为只表示"有活跃会话也照装"（不再隐含停 daemon）。
- daemon 启动时记录自己的 `codeHash`（已有，见 build-info）；空闲循环里比对磁盘 `codeHash`，不一致且**无活跃 turn + 无 in-flight 请求**时，走优雅自重启（沿用 `daemon-lifecycle` 的 startup-lock 串行化，避免和手动启动竞争）。
- 自重启对前台透明：`bridge.ts` 已有指数退避重连，daemon 换代窗口内的断连会自动恢复（复用现有重连，不新增协议）。

## 4. 影响文件 + 精确插入点

| 文件 | 改动 | 插入点 |

## 5. 分阶段实施（每阶段可独立闭环验证）

- **P1**：`install:global` 默认不停 daemon + `--restart-now` 开关。验收：装新版后 `agentbridge budget`/活跃会话仍在；`--restart-now` 才停。单测覆盖 arg 解析 + dry-run 文案。
- **P2**：daemon 漂移自检（只检测+log，不自重启）。验收：装新版后 `abg doctor` / 日志显示"detected newer bytes on disk, will restart when idle"。
- **P3**：安全点自重启（空闲+无 in-flight 才换代）。验收：E2E——会话 turn 进行中升级→不重启；turn 结束空闲→自动换代，新 `codeHash` 生效，前台自动重连无感。
- **P4**：文档（README + manual 双语）说明新升级语义。

## 6. 风险与边界

- 自重启必须**绝对避开活跃 turn**——宁可晚切换也不打断。判定从严：有任何 in-flight 请求 / 未结束 turn 一律不重启。
- startup-lock 串行化避免"自重启"和"用户手动 `agentbridge claude`"撞车。
- 不改 daemon 控制协议、不新增数据表（纯进程生命周期 + 脚本开关）。
- 仅 Personal/Trial 的本地 daemon 场景；Cluster/Production 的 systemd 部署不走 `install:global`，本方案不涉及（版型隔离）。
