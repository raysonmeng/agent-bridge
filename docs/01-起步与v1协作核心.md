# 01. 起步与 v1 协作核心（2026-03-21 ~ 03-25）

## 一句话定位

这是 AgentBridge 的**起点**——从一根能让 Claude 和 Codex 互相说话的"裸桥",到一套"会降噪、会让路、会分工"的协作核心。整个项目后续所有的多对、额度协调、workflow 编排,都站在这五天打下的地基上。

## 起点与背景:两个异构 AI agent 怎么协作

项目要解决的问题朴素而尖锐:一台机器上同时跑着 Claude Code 和 Codex,它们各有所长(Claude 强于架构、审查、规划;Codex 强于沙箱执行、复现、验证),但彼此听不见对方。让人在两个终端之间手动复制粘贴,既慢又丢上下文。

更麻烦的是这两个 agent **天生不对称**:

- Claude Code 通过 MCP 协议接入,可以注册工具、可以收 channel 推送通知;
- Codex 只有一个 app-server,没有"主动发消息给 Claude"的 API。

所以从第一天起,桥就不是一座对称的桥。它的设计哲学被一句话定死:**让 agent 自己做决策,bridge 只做极轻兜底**——不在桥里写复杂规则引擎或调度系统,而是用提示词契约引导 agent 行为,桥只做标记解析和最小化补救。这条哲学贯穿整个 v1,也成了项目的基因。

## 做了什么

**Initial commit:双向桥诞生。** v1.0 落地了最小可用形态:

- **双向消息桥** + **MCP 集成**(给 Claude 的 `reply` 工具);
- **Daemon 架构**——前台 `bridge.ts` + 后台 `daemon.ts` 分离,Claude Code 退出后 daemon 和 Codex 仍存活;
- **TUI 代理**——透明转发 Codex 流量,拦截 `agentMessage`;
- **`source` 字段**防止消息回环(桥永不把消息发回它的来源,这条不变量延续至今);
- **Channel 注入**——Codex 的话以 `<channel>` 标签出现在 Claude 会话里。

但 v1.0 是一座**裸桥**,自己列出的限制很诚实:全量转发无过滤、Codex 忙时无注入控制(只打 warning)、没有协作模式概念。

接下来用三个词、三条主线把裸桥升级成协作核心——**降噪、控回合、定角色**,分别回答"发什么 / 什么时候发 / 各自扮演谁":

- **v1.1 智能消息过滤(#3)——降噪,解决"发什么"。** 引入 **marker 协议**:`[IMPORTANT]`(需决策/review/阻塞/完成)、`[STATUS]`(阶段进展,可缓冲合并)、`[FYI]`(背景,默认丢弃)。桥按标记分流:IMPORTANT 立即转发,STATUS 进 summary buffer 合并,FYI 丢弃,无标记则兜底直通。配 `filtered`(默认)/ `full`(调试)两种模式。
- **v1.2 双向回合协调(#4)——控回合,解决"什么时候发"。** 承认两侧约束不对称,所以机制也不对称:Claude → Codex 用**硬信号**(基于 `turnInProgress`,补 `turnStarted` 事件上抛,推送 "⏳ Codex is working" / "✅ Codex finished",外加 busy reject 兜底);Codex → Claude 用**软性 attention window**(Claude 持有 floor 时给它一段思考窗口,期间 STATUS 缓冲、IMPORTANT 仍直通)。
- **v1.3 角色协作 + thinking patterns(#5)——定角色,解决"各自扮演谁"。** 默认角色分工(Claude:Reviewer/Planner/Hypothesis Challenger;Codex:Implementer/Executor/Reproducer)+ 三种协作思考模式(Independent Analysis & Convergence / Architect→Builder→Critic / Hypothesis→Experiment→Interpretation),按任务语义自动触发,并约定 `My independent view is:` / `I disagree on:` 等显式句式降低从众。

这三条主线最终被打包成一个**协作意识包**,在连接时自动注入双方("你不是一个人 + 怎么协作"),零用户配置——Claude 侧走 channel instructions,Codex 侧靠每次转发附带的 contract reminder 持续强化。

## 关键设计决策与为什么

**决策一:用 marker 协议让 agent 自己判断该不该转发,而不是在桥里写过滤引擎。**
理由很锋利:**agent 本身就是最强的语义理解模型**,应该让它在源头做决策,而不是在桥里做事后补救。规则引擎永远猜不准"这条消息重不重要",但写消息的那个 agent 知道。代价是 marker 是**软约束**——Codex 不一定每次都打对标(该 IMPORTANT 的可能标成 STATUS)。v1.1 明确接受这种不完美,因为目标是"体验明显变好",而非"形式上绝对正确"。这是项目第一次、也是反复出现的取舍姿态:**宁可要一个会偶尔漂移但极轻的契约,也不要一个精确但沉重的引擎。**

**决策二:回合协调"统一概念、不对称实现"。**
不强装对称是关键判断。Codex 有 `turnInProgress` 硬信号,就用硬 reject;Claude 没有等价的 turn 生命周期,就老实承认"测不准",改用启发式的 attention window。同时还克制地**拒绝做消息队列**——队列会引入排几条、何时 flush、用户取消怎么办等一堆状态语义,容易滑向 generalized coordination framework。v1.2 只要 "inform → respect floor → minimal reject"。

**决策三(本阶段最重要的演进信号):为兼容 API key 用户引入 push / pull 双模式(#2)。**
痛点真实:v1.0 完全依赖 Claude Code 的实验性 Channel(`notifications/claude/channel`),而它强制要求 `--dangerously-load-development-channels` + OAuth,把占比不小的 **API key 用户全挡在门外**。#2 的方案是双模式并行、自动检测切换:OAuth 用户走 **Channel Push**(实时),API key 用户走 **Tool Pull**(消息排队,Claude 主动调 `get_messages` 拉取),并用 `AGENTBRIDGE_MODE=push|pull|auto` 显式覆盖。两种模式共享同一套 daemon、`reply` 路径和 `BridgeMessage` 格式,唯一差异是传递方向。

## 踩的坑 / 教训:pull 模式为何后来被推翻

这是本阶段留给整个项目最有价值的一课,而且**坑是当时就预言到的**。

#2 的设计在"已知局限"里白纸黑字写着:pull 模式**没有机制"唤醒" Claude**,如果 Claude 不主动调 `get_messages`,消息会**静默堆积**——当时把它当作"支持 API key 认证的可接受折中"。

后来事实证明这个折中不可接受。今天代码里的判决很干脆(`src/claude-adapter.ts`、`src/unit-test/message-delivery.test.ts`):

> the old `AGENTBRIDGE_MODE=pull` delivery mode was removed: it could not wake an idle session and silently accumulated messages.

最终方案收敛成:**永远 push**,失败时落到一个由 `get_messages` 兜底排空的内存 fallback 队列;`AGENTBRIDGE_MODE` 环境变量被保留但忽略,只打一次性 warning(见项目 `CLAUDE.md` 的数据流不变量)。

教训有两层:其一,**"被动拉取 + 异步 agent"是反模式**——一个不会被唤醒的收件箱等于没有收件箱。其二,真正活下来的不是"双模式"这个形态,而是它内核里那个 `get_messages` 接口:它从"pull 模式的主通道"降格为"push 失败时的兜底队列",**机制活了,定位变了**。这正是一条健康的设计演进:先用一个看似合理的折中铺路,撞到现实后果断收敛到更简单、更可靠的单一通道。

## 产出:v1 留给项目的协作底座

这五天交付的不是一堆功能,而是一套**可被后续所有版本继承的协作能力底座**:

1. **一座可靠的双向桥** + 前台/后台分离的 daemon 架构 + `source` 防回环不变量——后续多对、重连、孤儿治理都长在这套骨架上;
2. **marker 协议 `[IMPORTANT]/[STATUS]/[FYI]`**——至今仍活在 `message-filter.ts` / `collaboration-content.ts`,并有专门测试守护,成为后续降噪分层的语义基础;
3. **回合协调的"硬信号 + 软窗口"范式**——`⏳ Codex is working` / `✅ Codex finished` 的交互至今是协作铁律;
4. **角色契约 + 三种思考模式**——"Claude 审查 / Codex 执行"的默认分工和 cross-review 习惯,从这里定型;
5. **一条贯穿全程的设计哲学**——agent 做决策、桥做轻兜底,以及"先折中、撞墙后果断收敛"的演进方法论。

更有象征意义的是:AgentBridge 本身正是用 Claude + Codex 通过这座桥协作造出来的(`Add project origin story` 提交)。它从第一天起就在**自举**。
